# archkit v1.9 ‚Äî CGR Test Gate + Deferred-Goal Proposals

**Status:** shipped (v1.9.0, 2026-06-04).
**Source:** CGR dogfood ‚Äî the relay loop's two soft spots: "done" meant "the agent said so," and follow-up work spotted mid-goal evaporated when the context cleared.
**Sibling docs:** [v1.8.md](v1.8.md) (the doctor / nextStep / observability line) and the v1.9.0 CHANGELOG entry.

## The two problems v1.9 exists to solve

v1.7‚Äìv1.8 built the Clear Goal Run loop: decompose a sprawling ask into one-goal-per-fresh-context units, relay between them with a single keystroke, and guard stopping with a Stop hook until the agent calls `archkit_goal_complete`. That loop trusts the agent at exactly two unguarded seams. v1.9 hardens both.

### 1. "Done" was a claim, not a proof

> The Stop-guard releases the moment the agent calls `archkit_goal_complete <slug>`. Nothing checked that the goal's *tests* actually pass ‚Äî completion was the agent asserting it was finished. A goal could archive green while the suite was red.

This is the same dead-end disease v1.8 named on the tool surface (a tool that returns "OK" when it did nothing), now at the goal layer: a `complete` that means "I believe I'm done" trains the operator to distrust the queue. The feedback loop needs an objective signal baked into the goal lifecycle ‚Äî completion should *provably* mean tests pass, not just that the agent says so.

### 2. Follow-up work spotted mid-goal evaporated

> Working a goal, the agent routinely notices adjacent work it should NOT do now ‚Äî "out of scope for this PR," "follow-up: wire up retries," "we'll handle pagination in a separate goal." In a one-goal-per-context discipline, that observation is correct (don't derail the active goal) but lossy: the next `/clear` wipes it.

The CGR principle is "one goal per fresh context." Its cost is that everything *not* the active goal is forgotten at the context boundary. v1.9 captures deferrals as **proposals** ‚Äî stashed, reviewed, and promoted into the queue on the operator's say-so ‚Äî instead of lost.

## v1.9 work items

### A. CGR test gate ‚Äî auto-detected `verify-command`

`archkit_goal_intake` now detects the project's test command and stamps it onto every goal as `verify-command`.

- **Detection** ([`src/lib/test-runner.mjs`](../../src/lib/test-runner.mjs) ‚Üí `detectTestCommand`): reads `package.json` ‚Üí `scripts.test`, then picks the runner from the lockfile (`pnpm-lock.yaml` ‚Üí `pnpm`, `yarn.lock` ‚Üí `yarn`, `bun.lockb` ‚Üí `bun`, else `npm`). Best-effort and Node-first, because archkit is a Node tool.
- **Graceful skip**: returns `null` when there's no real test script ‚Äî no `package.json`, no `scripts.test`, or npm's `no test specified` placeholder (matched by `NPM_PLACEHOLDER`). Projects without tests aren't blocked on a command that can't run; intake just reports "no test gate."
- **Override**: a goal can carry its own `verifyCommand` / `verify-command` field; intake only fills in the detected default when one isn't already set ([`src/commands/goal.mjs`](../../src/commands/goal.mjs) ‚Üí `if (!g.verifyCommand && detected)`).

### B. CGR test gate ‚Äî hard gate on `archkit_goal_complete`

Completion re-runs the `verify-command` and **refuses to complete a goal whose tests are red** ([`src/commands/goal.mjs`](../../src/commands/goal.mjs), the `runTests` block):

- Command can't be launched (runner missing, spawn fails) ‚Üí throws `test_gate_unrunnable`.
- Tests red (non-zero exit, or timed out) ‚Üí throws `test_gate_failed`, carrying the failing `outputTail` (last ~2000 chars of combined stdout+stderr) and a remediation suggestion ("Fix the failing tests, then retry `archkit_goal_complete <slug>`").
- Tests green ‚Üí stamps `tests-passed: true`, `tests-command`, and `tests-at` (ISO date) onto the archived goal as objective completion evidence.

The escape hatch for a genuinely-obsolete goal is **`archkit_goal_abandon`**, not completion ‚Äî abandon archives with status `abandoned` and never pretends the work was finished.

### C. CGR test gate ‚Äî cheap preview on `archkit_goal_verify`

`archkit_goal_verify` runs the same command as a **non-authoritative dry run** and folds the result into the goal's `clean` signal, so the agent sees green/red (and keeps working on red) *before* it calls complete. `runTests()` ([`src/lib/test-runner.mjs`](../../src/lib/test-runner.mjs)) never throws ‚Äî a spawn failure is reported as `ran: false` so callers degrade gracefully rather than crashing the verify path.

### D. Deferred-goal proposals ‚Äî defer / promote / dismiss (3 new MCP tools)

Brings the tool count from 25 to **28**:

- **`archkit_goal_defer`** (26th tool added): stashes a follow-up the agent notices mid-session as a **proposed** goal in `.arch/goals/proposed/` ‚Äî without derailing the active goal or touching the queue. Richer than a Stop-hook draft because the agent supplies a real title + exit-criteria. Dedupes by title hash.
- **`archkit_goal_promote`** (27th): the "confirm" half ‚Äî turns selected proposals (`hashes:[...]` or `all:true`) into planned goals the CGR queue picks up, and removes them from `proposed/`.
- **`archkit_goal_dismiss`** (28th): the "reject" half ‚Äî discards proposals without promoting them.

Library: [`src/lib/goals.mjs`](../../src/lib/goals.mjs) (`proposedDir`, the propose/promote/dismiss helpers).

### E. Deferred-goal proposals ‚Äî `/mcp__archkit__goal_review` prompt

A new MCP prompt ([`src/mcp/prompts.mjs`](../../src/mcp/prompts.mjs)) lists pending proposals and drives an `AskUserQuestion` multi-select so the operator picks which to promote vs. dismiss in one pass. Anything neither promoted nor dismissed stays pending. Like the other relay prompts, it's user-typed ‚Äî the agent can't trigger it (it pairs with the human's review moment).

### F. Deferred-goal proposals ‚Äî Stop-hook auto-drafting

The Stop hook scans each assistant turn for explicit deferral language and auto-drafts a proposal to `.arch/goals/proposed/<hash>.json` ([`bin/archkit-stop-hook.mjs`](../../bin/archkit-stop-hook.mjs) ‚Üí `detectDeferredGoals`):

- **Detector** ([`src/lib/goal-detector.mjs`](../../src/lib/goal-detector.mjs)): six case-insensitive patterns ‚Äî explicit follow-up/TODO labels, "out of scope for this/now," "in a separate goal/PR," "belongs in its own follow-up," "punting on / circle back / revisit later," and "we'll handle that later."
- **High-precision-over-recall**, deliberately mirroring [`decision-detector.mjs`](../../src/lib/decision-detector.mjs): questions and exploratory "should we‚Ä¶?" phrasing are filtered out (`looksLikeQuestion`), and titles thinner than 8 chars are dropped. False positives spam the queue and erode trust faster than a missed follow-up hurts.
- **Cross-turn dedup** by `sha1(titleHint::matchedText)`, so the same deferral mentioned across turns drafts once.

The auto-draft is intentionally the *thin* path (title hint + excerpt). The *rich* path is `archkit_goal_defer`, where the agent writes a real title and exit-criteria.

## Non-goals for v1.9

- **New domain verbs.** Like v1.8, this is loop-hardening, not new capability ‚Äî the test gate and proposals both reinforce the existing CGR workflow.
- **Auto-promotion of proposals.** A drafted proposal never becomes a planned goal without a human (`promote` / `goal_review`). Same human-gated posture as proposed ADRs (v1.6) and proposed BANs (v1.8).
- **Non-Node test detection.** `verify-command` detection is Node-ecosystem-first. Other stacks set `verifyCommand` per goal explicitly; the gate runs whatever command is set, language-agnostically.
- **Schema migration.** Every field (`verify-command`, `tests-passed`/`-command`/`-at`, `proposed/`) is additive. Projects without a test script skip the gate; the propose/review flow is opt-in.

## Open questions

1. **Detector calibration on real corpora.** The deferral patterns are co-tuned with their tests (same caveat the v1.6 decision-detector carries). Field precision/recall on real sessions is the follow-up.
2. **Per-goal timeout.** `runTests` defaults to a 180s timeout. Long suites may need a per-goal `verify-timeout` knob rather than a global default.
3. **Proposal staleness.** Proposals accumulate in `.arch/goals/proposed/` until reviewed. Should `archkit doctor` (v1.8) surface a "N pending goal proposals ‚Äî run /mcp__archkit__goal_review" check the way it surfaces pending ADRs?

## Release shape

v1.9.0 = A + B + C (test gate) + D + E + F (deferred-goal proposals). Shipped as one release because they're the two halves of the same fix: harden the CGR loop's entry (intake bakes the gate) and exit (complete enforces it), and stop losing the work the loop's own discipline forces you to defer.

## Validation

New suites `tests/test-gate/` (verify-command detection + the hard completion gate) and `tests/goal-proposals/` (defer/promote/dismiss + the deferral-language detector). `tests/mcp-server/` tool-count assertion updated 25 ‚Üí 28; `tests/silent-success-audit/` and `tests/stop-hook/` updated for the new tools and auto-drafting. Same dogfood pattern as prior releases: run the CGR loop on a real multi-goal ask, confirm the gate blocks a red goal and that mid-goal deferrals land as reviewable proposals.
