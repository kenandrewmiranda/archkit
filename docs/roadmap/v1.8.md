# archkit v1.8 — Dead-end Indicators, Workflow Logistic Gauge, One-Function-One-Goal

**Status:** scoping doc, not yet committed work.
**Source:** conference feedback (2026-05-25) from working developers using archkit and similar tooling.
**Sibling docs:** v1.7's CHANGELOG entry for the quick-win subset of these themes.

## The three problems v1.8 exists to solve

These came from a single conversation but they're the same disease at different layers.

### 1. Dead-end indicators

> "A function isn't fully fleshed out and the user doesn't get any feedback or silently fails in the background. The user hits an error but there is no resolution for the user OR for the developer to understand what happened. So partial logging on the developer side and communication on the user side."

A tool that returns "OK" when it actually did nothing is worse than one that errors loudly. It trains users to ignore the tool. The arch-poly review-staged silent-zero bug (fixed in v1.6.5) and the boundary-check "rules: 0 → pass: true" gap (partially closed in v1.7 with the `hint` field) are the same shape: the success signal didn't carry "what got checked." Both sides — the **agent** that decides what to do next and the **human** debugging "why didn't this catch it?" — need observability.

Concrete v1.7 starts: `archkit_boundary_check` now reports `unappliedRules`, `archkit_resolve_preflight` reports `requiredReadingNote` + `skillCatalogSize`. v1.8 extends the principle to every tool.

### 2. Workflow logistic gauge

> "Developers will find the most optimal approach, but the user may not align or understand the logical approach so there needs to be logistic checks on when a new function is added and how it relates to the greater workflow."

Optimal-for-the-developer ≠ usable-for-the-caller. archkit has 19 MCP tools as of v1.7; the agent (and the human reading the agent's output) has to know which to call when. Today the tool *descriptions* explain when-to-use, but the tool *outputs* rarely explain what-comes-after. v1.7 quick-wins added a `nextStep` field to `archkit_boundary_check` and `archkit_resolve_preflight`. v1.8 standardizes this across every tool — and adds a meta-check ("does the response actually tell the agent what to do next?") to the tool registration.

### 3. One function, one goal

> "Each workflow in the app/program should always have a goal that it accomplishes. If it solves multiple goals or no goal, then it's not following through the principle of one function one goal."

The archetype anti-pattern: `archkit_resolve_preflight` now does five distinct jobs (feature lookup + git history + pending gotchas + drift + required reading). It's cohesive ("everything you need to know before touching this feature") but the response is large, and a caller wanting just "what skills should I read" pays for git-log walks they don't need. Two of those five — `requiredReading` and `pendingGotchas` — have plausible standalone consumers.

This is the hardest of the three problems to act on because it requires usage telemetry the project doesn't have yet. **v1.8 should add the telemetry, then decompose based on data.**

## v1.8 work items

### A. Silent-success audit (every tool)

For each of the 19 MCP tools, classify the response surface:

| Outcome class | Today's signal | Desired signal |
|---|---|---|
| Real success | `pass: true` or domain field populated | unchanged |
| Checked, found nothing | `pass: true`, empty array | + reason field explaining *why* it's empty |
| Couldn't check (missing input) | error or empty | + error code + recovery suggestion |
| Partial check (some inputs invalid) | silent skip | + `skipped` array with reasons |

Touch: `src/mcp/tools.mjs` schema + every `run*Json` runner. New runner-level convention: return `{ checked: {...counts...}, ...domain }` so the agent always knows what scope was scanned.

Deliverable: a `tests/silent-success-audit/run.mjs` that asserts every tool's response includes either a populated result OR an explanation of why it's empty. Run as part of CI. Block PRs that add a tool without the explanation field.

### B. nextStep convention (every tool)

Promote v1.7's pattern (`archkit_boundary_check.nextStep`, `archkit_resolve_preflight.nextStep`, `archkit_goal_intake.nextStep`, `archkit_goal_complete.nextGoal`) to a hard contract.

Spec:
- Every `run*Json` returns a `nextStep: string` field
- Format: imperative-mood, ≤140 chars, name the *next tool* or *next user action* if applicable
- Validated in CI by the same audit suite

Examples of good and bad:
- Good: `archkit_goal_complete` → `"Run /clear and paste nextGoal.payload after /goal"`
- Bad (current `archkit_drift`): no field — agent has to infer "fix the drift" from the findings array

Touch: every runner + the audit suite.

### C. Workflow logistic gauge — the `archkit doctor` command

A meta-tool that walks the full archkit surface and reports dead-ends:
- Skills with zero gotchas in `.arch/skills/`
- BAN rules in BOUNDARIES.md that no file currently in the repo matches (warning, not error — could be future-protecting)
- Active goals in `.arch/goals/` with vacuous exit-criteria (e.g. "ship it") or no `required-reading`
- INDEX.md entries pointing at paths that don't exist (already drift, but `doctor` aggregates)
- MCP tools whose description was last edited >90 days ago (drift between tool behavior and description)

Output: `{ checks: [...], warnings: [...], blockers: [...], nextStep }`. Wraps `warmup` + `drift` + new checks. Surfaced as both CLI (`archkit doctor`) and MCP (`archkit_doctor`).

**Why `doctor` and not "more warmup checks"?** Because `warmup` runs at session start and is structural ("can I trust .arch/ at all?"). `doctor` runs on demand and is *intent-checking* ("is the rich content of .arch/ actually load-bearing?"). Different question, different cadence.

### D. One-function-one-goal — telemetry, then decompose

Add lightweight usage telemetry (opt-in, local-only) to the MCP server:
- For each tool call, record: tool name, which response fields the caller appears to use (heuristic: which fields appear in subsequent tool calls as input)
- Aggregate in `.arch/.usage-stats.json` (gitignored)
- New CLI: `archkit stats --tool-usage` → table of "field X in tool Y is referenced by N% of subsequent calls"

After 30 days of data on a real project:
- If `archkit_resolve_preflight.requiredReading` is consumed by 80% of callers but `pendingGotchas` only 15%, that's evidence for decomposition into `archkit_resolve_skills_for_feature` (cheap, single-purpose) + `archkit_resolve_preflight` (kitchen-sink, opt-in)
- If `pendingGotchas` is rarely consumed at all, that's evidence it should be a separate `archkit_gotcha_pending_for_feature` callable on demand

**No decomposition without data.** Splitting tools speculatively creates more surface area to maintain. Wait for telemetry.

### E. Stretch — Dev-side logging fidelity

Today archkit logs to stderr in human-readable form (and to stdout as JSON when invoked in --agent mode). For the "developer hits an error and can't understand what happened" half of the dead-end problem:
- Add an `ARCHKIT_LOG_LEVEL=debug` env that streams structured JSON-line logs of every tool's internal decisions to a file (`.arch/.archkit.log`, gitignored)
- Include: which input fields were validated, which code paths fired, which optional features were skipped and why
- Surface a `--debug-log <path>` CLI flag for ad-hoc captures
- `archkit doctor` (item C) can include "recent errors in .archkit.log" as a check

This is the developer-side counterpart to the user-side `nextStep` work.

## Non-goals for v1.8

- New domain capability. v1.7 added CGR + boundary-check; v1.8 is hygiene + observability, not new verbs.
- Schema migration. The `nextStep` field is additive; nothing breaks.
- Decomposing `preflight` ahead of telemetry. See item D.

## Open questions

1. **Telemetry consent model.** Opt-in only? Or default-on with redacted/aggregate-only data? Conference feedback emphasized user-side communication — telemetry that the user can't see or correct fails that principle.
2. **`doctor` vs `warmup --deep`.** Possible overlap. The split above is defensible but worth pressure-testing once the doctor checks are concrete.
3. **`nextStep` for multi-step workflows.** When one tool's `nextStep` is "call tool X with these args," should we structure that machine-readably (`nextTool: { name, args }`) so agents can chain without parsing prose? Arguably yes, and arguably that's a separate `archkit_workflow` concept worth scoping.

## Release shape

v1.8.0 = A + B + C (audit + nextStep convention + doctor). Ship as one release because they're the same principle expressed three ways.
v1.8.1 = D (telemetry) once A/B are field-tested.
v1.9 = decomposition decisions based on D's data.

## Validation

Same pattern as v1.7: dogfood on a sibling project (probably arch-poly or arch-market, whichever is most active when v1.8 is ready), capture the "did the tool tell you what to do next?" question explicitly, iterate.
