# archkit v1.3.0 — Agent-First Depth Release

**Date:** 2026-04-17
**Status:** Draft
**Source issues:** kenandrewmiranda/archkit#19, #20, #21

---

## 1. Overview

archkit's target audience is AI agents, not humans. AI agents ship surface-level solutions — code that compiles but skips production-ready depth. v1.3 is the "depth" release: every change either gives the agent richer context the static files can't carry, catches AI's surface-level defaults before they ship, or makes the protocol actually enforceable at runtime.

| # | Change | Issue |
|---|--------|-------|
| 1 | `preflight` becomes live (git history + scoped gotchas + scoped drift) | #20 |
| 2 | `scaffold` becomes live (generates files with `AGENT-VALIDATION` gates, dry-run by default) | #20 |
| 3 | `plan` + `context` soft-deprecated → remove in v2.0.0 | #20 |
| 4 | Production-readiness checks added to `review` (floating promises, mock data, dead handlers, untracked TODOs, incomplete skeletons) | #19 (reframed) |
| 5 | Suppression with required substantive justification | #19 |
| 6 | `archkit-protocol` skill wraps real `preflight` call with structured output | #21 |
| 7 | `init --install-hooks --claude` installs Claude Code `PreToolUse` hook | #21 |
| 8 | Terse-by-default output across all new surfaces; `--verbose` opt-in | architectural |

**Out of scope (deferred):**
- Polyglot verify-wiring scanner (separate issue #22)
- Suppression of architecture rules — only generic and production-readiness rules support suppression in v1.3 (architecture violations should always require code change, not a comment)

**Version:** v1.3.0 — additive features, soft deprecations, no breaking changes. Removal of `plan`/`context` deferred to v2.0.0.

## 2. Architectural principle: agent-first

Every design decision in this release uses one lens: *does this help the AI agent ship production-quality code, or does it duplicate what mature human-facing tools already do?*

This means:
- **Default output is terse.** Agents need actionable lines, not paragraphs. `--verbose` opt-in for explanations.
- **Checks are deep.** Catch the patterns AI defaults to that humans wouldn't write (floating promises, mock data leftovers, dead error handlers).
- **Generated code includes validation gates.** Skeletons embed `AGENT-VALIDATION` blocks that `review` enforces. AI cannot lazy-ship a generated stub.
- **Live runtime data over static restatement.** `preflight` and `scaffold` become genuine views over real state, not parrots of the .arch/ files.

## 3. Live `preflight`

### 3.1 Invocation

`archkit resolve preflight <feature> <layer>` (unchanged surface, new behavior)

### 3.2 What it returns

A live runtime view scoped to the feature/layer being touched:

1. **Recent commits** — `git log -5 --format=... -- <basePath>` filtered to the feature's basePath
2. **Pending gotchas** — `.arch/gotcha-proposals/*.json` filtered to proposals where `skill` matches a package used by the feature (cross-referenced via INDEX.md)
3. **Drift findings** — current drift checks filtered by feature's basePath
4. **Last-touched timestamp** — derived from most recent commit

### 3.3 Default output (terse)

```
@kalshi/streams — last touched 2 days ago (commit a1b2c3d, Ken)
2 pending gotchas: aiosqlite-txn-flush, sqlite-alter-check
3 drift findings in scope: missing-file kalshi_ws.py, missing-file espn.py, missing-file draftkings.py
```

### 3.4 Verbose output (`--verbose`)

Same data with full content: full git log entries (subject + date + author), full gotcha proposal text (skill + WRONG/RIGHT/WHY), full drift detail per finding.

### 3.5 JSON output (`--json`)

```json
{
  "feature": "kalshi",
  "layer": "streams",
  "lastTouched": { "commit": "a1b2c3d", "author": "Ken", "date": "2026-04-15", "daysAgo": 2 },
  "recentCommits": [
    { "hash": "a1b2c3d", "subject": "fix: kalshi reconnect logic", "date": "2026-04-15", "author": "Ken" }
  ],
  "pendingGotchas": [
    { "hash": "abc123def456", "skill": "aiosqlite", "wrong": "...", "right": "...", "why": "..." }
  ],
  "driftFindings": [
    { "type": "missing-file", "id": "kalshi", "detail": "..." }
  ],
  "passWithoutAction": false,
  "gitAvailable": true
}
```

`passWithoutAction` is `true` when all three arrays are empty — the green light for the agent.

### 3.6 Implementation surface

`src/commands/resolve/preflight.mjs` (currently small) gets restructured:

1. Parse INDEX.md to get feature's basePath + skills (existing parsers.mjs logic)
2. Run `git log -5 --format=... -- <basePath>` for recent commits
3. Read `.arch/gotcha-proposals/*.json`, filter where `skill` matches feature's skills
4. Run drift checks but filter findings by feature's basePath (reuses fixed drift.mjs)
5. Compose into JSON shape; renderer converts to terse/verbose human output

### 3.7 Error handling

- Feature not found in INDEX.md → exit 1 with `{"error":"unknown_feature","feature":"X","valid":[...]}`
- Layer not found in feature's cluster graph → exit 1 with `{"error":"unknown_layer","layer":"X","valid":[...]}`
- Not a git repo → continues without git data; `gitAvailable: false` in JSON; recentCommits omitted
- `git log` errors (corrupt repo, etc.) → silent fallback to no git data

### 3.8 Performance target

<500ms total. Git log is the slowest step (~50-200ms). Other reads are cheap.

## 4. Live `scaffold`

### 4.1 Invocation

- `archkit resolve scaffold <feature>` → dry-run (default, safe — never writes)
- `archkit resolve scaffold <feature> --apply` → writes files
- `archkit resolve scaffold <feature> --apply --overwrite` → writes files even if targets exist (matches existing `init --overwrite` pattern)

### 4.2 What it generates

Per feature, based on the project's app type from `APP_TYPES`:

1. `.arch/clusters/<feature>.graph` — populated cluster graph following the app type's pattern
2. `.arch/INDEX.md` — appended with new node entry
3. Source skeleton files following `appType.folderConv`:
   - SaaS: `<feature>.controller.ts`, `<feature>.service.ts`, `<feature>.repository.ts`, `<feature>.types.ts`
   - Mobile: `<Feature>Screen.tsx`, `use<Feature>.ts`, `<feature>.service.ts`
   - AI: `<feature>.chain.ts`, `<feature>.prompt.ts`
   - Other app types: per their `folderConv` patterns
4. Imports wired between generated files (controller → service → repository)

### 4.3 Skeleton file structure

Every generated stub starts with two blocks:

**`AGENT-VALIDATION` block (enforced by `review`):**

The block uses the comment style of the target language (`//` for JS/TS, `#` for Python, `--` for SQL, etc.) — `src/lib/skeleton-renderer.mjs` selects the comment prefix based on file extension.

```typescript
// ┌─ AGENT-VALIDATION (required before completion) ────────────────
// │ Skeleton generated by archkit scaffold. Validate against current
// │ official docs BEFORE filling in implementation. Tick each box
// │ ([x]) when verified or remove the entire block to confirm done.
// │
// │ [ ] Hono routing + Context API → https://hono.dev/api/context
// │ [ ] Project's $err types match what this controller throws
// │ [ ] Project's $zod schema pattern for request validation
// └─────────────────────────────────────────────────────────────────
```

**`AGENT-INSTRUCTIONS` block (guidance):**

```typescript
// AGENT-INSTRUCTIONS: Replace this stub with the real controller logic.
//   - Validate request inputs via shared $zod schemas
//   - Delegate to <feature>Service for business logic
//   - Throw $err types instead of generic Error
```

Then the actual stub: imports + minimal exports + section markers.

### 4.4 Doc URLs in validation block

URLs come from `src/data/package-docs.mjs` based on the detected stack. If a package has no entry, that line is omitted from the validation block.

### 4.5 Verbose validation block

With `--verbose`, each validation line gains a "why this matters" sub-line:

```typescript
// │ [ ] Hono routing + Context API → https://hono.dev/api/context
// │     (Hono v4 changed Context.req.json() return type — verify your version)
```

### 4.6 Dry-run output (default, terse)

```
Would create 5 files:
  .arch/clusters/notifications.graph     (cluster graph)
  src/features/notifications/notifications.controller.ts
  src/features/notifications/notifications.service.ts
  src/features/notifications/notifications.repository.ts
  src/features/notifications/notifications.types.ts
Would update: .arch/INDEX.md (+1 node entry)

Run with --apply to write.
```

### 4.7 Dry-run JSON output

```json
{
  "feature": "notifications",
  "appType": "saas",
  "wouldCreate": [
    { "path": ".arch/clusters/notifications.graph", "type": "cluster-graph", "size": 240 },
    { "path": "src/features/notifications/notifications.controller.ts", "type": "skeleton-controller", "size": 380 }
  ],
  "wouldUpdate": [
    { "path": ".arch/INDEX.md", "diff": "+@notifications = [notifications] → src/features/notifications/" }
  ]
}
```

With `--apply`, fields rename to `created` and `updated`.

### 4.8 Conflict handling

- Target file exists → skip with `{"path":"...","skipped":"already exists"}` in output. NEVER overwrite. NEVER prompt.
- Explicit `--overwrite` flag forces writes for users who really want it.
- This is the agent-safe default — agents detect skips in JSON and decide whether to delete + re-run, or work around.

### 4.9 Implementation surface

| File | Action |
|------|--------|
| `src/commands/resolve/scaffold.mjs` | RESTRUCTURE — add file generation logic, dry-run handling, --apply gate |
| `src/data/skeleton-templates.mjs` | NEW — exports `SKELETONS = { saas: { controller: "...", ... }, mobile: {...}, ... }` per app type per layer |
| `src/lib/skeleton-renderer.mjs` | NEW — substitutes `{cluster}`, `{layer}`, `{Layer}`, `{ClusterName}` etc. into template strings |
| `src/data/package-docs.mjs` | EXTEND — ensure all packages from `APP_TYPES.defaultStack` have doc URLs |

### 4.10 Non-goals

- No test stub generation (too many ways to be wrong about test framework choice)
- No route registration in app.ts/main.ts (too many ways to mangle existing files)
- No cross-feature shared-interface stubs

## 5. `review` additions

### 5.1 New production-readiness checks

Four new checks live in `src/commands/review/production-checks.mjs`. Each is callable from the existing review pipeline alongside current checks.

#### 5.1.1 `floating-promise`

**Pattern:** Async function called without `await`, no `.then()`, no `.catch()`, not assigned, not returned.

**Examples flagged:**
- `someAsyncFn();` on its own line
- `arr.map(asyncFn)` (Promise.all not used; result discarded)

**Severity:** `error`

**Skip rules:**
- Explicit fire-and-forget marker: `void someAsyncFn();`
- File path includes `test`, `spec`, `script`, `__tests__`

#### 5.1.2 `mock-data-leftover`

**Pattern:** Markers AI uses for placeholder data in non-test files.

**Triggers:**
- Comments: `// mock data`, `// fake data`, `// test data`, `// dummy data`
- `Math.random()` outside game/scientific contexts
- Hardcoded fake names: `"John Doe"`, `"Jane Doe"`, `"Test User"`, `foo@bar.com`, `example.com`
- Generated-looking UUIDs: `00000000-0000-0000-0000-000000000000`, `11111111-...`, etc.

**Severity:** `error`

**Skip rules:**
- File path includes `test`, `spec`, `fixture`, `mock`, `stub`, `__mocks__`

#### 5.1.3 `dead-error-handler`

**Pattern:** `catch` block that does nothing useful.

**Triggers:**
- `catch (e) {}` — empty
- `catch (e) { console.log(e); }` — log-and-swallow
- `catch (e) { console.error(e); }` — log-and-swallow

**Severity:** `error` for above patterns; `warning` for `catch (_) {}` (intentional ignore — flag less aggressively)

**Skip rules:**
- catch block re-throws (`throw e`, `throw new ...`)
- catch block returns an error type
- catch block calls a project logger (`log.error(e)`, `logger.error(e)` — uses logger import detection)
- catch block calls a known error handler function

#### 5.1.4 `untracked-todo`

**Pattern:** TODO/FIXME/XXX without a tracking reference.

**Examples:**
- Flagged: `// TODO: handle errors`
- OK: `// TODO(KAL-42): handle errors`
- OK: `// TODO(2026-05-01): handle errors`
- OK: `// TODO(@kenmiranda): handle errors`
- OK: `// TODO(#42): handle errors`

**Severity:** `warning`

**Tracking patterns recognized:**
- Ticket: `TODO(JIRA-123)`, `TODO(KAL-42)`, `TODO(#42)`
- Date: `TODO(2026-05-01)`, `TODO(2026-Q2)`
- Owner: `TODO(@kenmiranda)`, `TODO(@team-platform)`

#### 5.1.5 `incomplete-skeleton`

**Lives in:** `src/commands/review/completeness-checks.mjs` (extends existing file)

**Pattern:** Source file contains `AGENT-VALIDATION (required` block AND has any unchecked `[ ]` boxes.

**Behavior:**
- If block found AND any `[ ]` present → finding: `[error:incomplete-skeleton] Unvalidated skeleton — verify against docs and tick boxes (or remove the block)`
- If block fully ticked (all `[x]`) OR removed entirely → no finding

**Severity:** `error`

**Why this matters:** This is the enforcement half of the scaffold validation gate. Scaffold creates the gate, review enforces it. Together they prevent AI from lazy-shipping generated skeletons.

### 5.2 Suppression mechanism

#### 5.2.1 Syntax

`// archkit: ignore <rule-id> — <reason>` on the line above OR same line as the violation.

#### 5.2.2 Reason validation

Reason is mandatory and substantive. archkit rejects shallow reasons matching this case-insensitive blocklist:

```
n/a, none, see comment, see above, see below, fixed, ok, false positive,
todo, will fix, ignore, skip, intentional, on purpose, comment
```

#### 5.2.3 Weak suppression handling

If reason matches blocklist, suppression doesn't apply AND a new finding is emitted:

```
[error:weak-suppression] Suppression reason "<reason>" is too vague — explain why this code is correct.
```

#### 5.2.4 What can be suppressed

- All generic checks (db-efficiency, cache-queue, api patterns, etc.)
- All four new production-readiness checks (floating-promise, mock-data-leftover, dead-error-handler, untracked-todo)
- `incomplete-skeleton` check

**What cannot be suppressed:** Architecture rules (BOUNDARIES violations, layer-hierarchy violations, reserved-word violations). These represent intentional design constraints — code change required, not a comment.

#### 5.2.5 Examples

```javascript
// archkit: ignore db-efficiency/select-star — small lookup table, columns vary per consumer
const trades = db.query("SELECT * FROM trades WHERE exit_price IS NULL");  // ✅

// archkit: ignore dead-error-handler — caller intentionally ignores network failures, logged elsewhere
try { await ping(); } catch (e) {}  // ✅

// archkit: ignore floating-promise — fire-and-forget telemetry, batched and retried in worker
recordEvent("page-view");  // ✅

// archkit: ignore floating-promise — fixed
asyncCall();  // ❌ rejected (weak reason)
```

#### 5.2.6 Project-level suppression

`.arch/review.yaml` for project-wide rule disabling:

```yaml
disabled_rules:
  - db-efficiency/select-star  # lifetime-max-100-rows tables; not worth annotating each
disabled_for_paths:
  - rule: untracked-todo
    paths: ["docs/**", "scripts/**"]
```

Project-level disables don't require a per-rule reason in the file (the file IS the reason — committed to repo, reviewable in PR).

### 5.3 JSON output additions

`review --json` output gains:

```json
{
  "files": 42,
  "errors": 3,
  "warnings": 8,
  "findings": { "...": [...] },
  "suppressed": [
    { "file": "bot/storage/db.py", "line": 250, "ruleId": "db-efficiency/select-star", "reason": "..." }
  ],
  "weakSuppressions": [
    { "file": "bot/api.py", "line": 42, "ruleId": "floating-promise", "reason": "fixed" }
  ]
}
```

## 6. Skill + hook integration

### 6.1 `archkit-protocol` skill becomes substantive

The current skill is documentation prose. The new skill is a thin wrapper that runs the relevant archkit command and returns structured output.

**New skill content:**

```markdown
---
name: archkit-protocol
description: Use BEFORE editing or creating files in an archkit-governed project. Returns live state for the targeted feature/layer — recent commits, pending gotcha proposals, drift findings in scope. Skipping this means writing code blind to recent changes, known issues, and architectural drift.
---

# archkit Protocol

Before editing files in this project, invoke this skill with the feature
and layer you're about to touch.

## Required step

Run: `archkit resolve preflight <feature> <layer> --json`

Parse the output for:
- `recentCommits`: was someone else just here? Coordinate or merge.
- `pendingGotchas`: are there reviewed-but-not-yet-merged warnings about this code?
- `driftFindings`: is the .arch/ map already inconsistent with the code? Don't add more divergence.

If `passWithoutAction` is `true`, proceed. Otherwise, address the surfaced
issues OR explicitly justify ignoring them in your work plan.

## Why this exists

archkit's static `.arch/` files describe intent. This skill surfaces *current
state* — what changed recently, what's broken right now, what's queued for
review. Without it you're working from a snapshot that may be days old.
```

The skill's value is now concrete: it returns data agents can't get from reading static files.

### 6.2 Claude Code `PreToolUse` hook

**New invocation forms for the existing `init --install-hooks`:**

- `archkit init --install-hooks` (existing) → installs `.git/hooks/pre-commit`
- `archkit init --install-hooks --claude` (new) → installs both git hook AND Claude hook
- `archkit init --install-hooks --claude-only` (new) → only Claude hook, skip git

### 6.3 Claude hook implementation

The hook is too dense for inline shell. Ships as a standalone Node script: `bin/archkit-claude-hook.mjs`. The `package.json` `bin` field gets a new entry (`"archkit-claude-hook": "./bin/archkit-claude-hook.mjs"`) so it's available on PATH after install, same mechanism as the main `archkit` binary.

`.claude/settings.json` snippet that gets installed:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "archkit-claude-hook \"$TOOL_INPUT_PATH\"",
            "timeout": 3000
          }
        ]
      }
    ]
  }
}
```

`bin/archkit-claude-hook.mjs` does:
1. Take file path as arg
2. Derive feature/layer from path (regex against common patterns: `src/features/<feature>/<layer>`, `bot/<layer>/<feature>`, etc.)
3. If derivable, run `archkit resolve preflight <feature> <layer> --json`
4. If `passWithoutAction` is true → exit 0 silently
5. Else → write summary to stderr, exit 2 (Claude Code surfaces stderr to agent)
6. If feature/layer not derivable from path → exit 0 silently (don't block unrelated edits)

### 6.4 Hook safety

- **3-second timeout** — if archkit hangs, hook silently passes
- **Exit 0 always for unknown paths** — don't block edits outside known feature directories
- **Exit 2 for surfaced issues** — informational, not fatal; Claude Code surfaces this to the agent
- **Best-effort, not hard-block** — agent retains judgment

### 6.5 `.claude/settings.json` merge strategy

If file already exists, archkit reads it, adds the hook to the `PreToolUse` array (preserving existing hooks), writes back. Same conflict-handling pattern as the git hook installer.

If existing settings have a conflicting `archkit-claude-hook` entry, returns `{"status":"existing-claude-hook","suggested_append":"..."}` — never overwrites blindly.

### 6.6 JSON output for `init --install-hooks --claude`

```json
{
  "status": "installed",
  "git_hook": ".git/hooks/pre-commit",
  "claude_hook": ".claude/settings.json",
  "claude_settings_action": "merged"
}
```

### 6.7 Implementation surface

| File | Action |
|------|--------|
| `bin/archkit-claude-hook.mjs` | NEW — standalone script invoked by Claude hook |
| `src/commands/init.mjs` | EXTEND `--install-hooks` — add `--claude` and `--claude-only` flags |
| `src/data/skill-templates.mjs` | NEW — exports `ARCHKIT_PROTOCOL_SKILL` template |
| `src/lib/claude-settings.mjs` | NEW — read/merge/write `.claude/settings.json` safely |

### 6.8 What this does NOT do

- Does NOT block edits. Hook is informational only.
- Does NOT enforce that the agent must read the output. Agent can ignore it. But "ignore" now requires a deliberate choice instead of missing context.
- Does NOT modify the agent's prompt or inject context.

This is honest enforcement: protocol fires automatically, agent retains judgment. We've moved from "non-negotiable in prose, optional in practice" to "automatic in practice, informational by design."

## 7. Verbose flag standardization

### 7.1 Default behavior across new surfaces

Terse output. The bare minimum an agent needs to act.

### 7.2 `--verbose` / `-v` opt-in

Adds rationale, "why this matters", suggested fixes, doc URLs, full context.

### 7.3 Applies to

- `archkit resolve preflight` — terse 3-line summary; verbose adds full git/gotcha/drift detail
- `archkit resolve scaffold` — terse file count + paths; verbose adds full file tree + INDEX.md diff + per-template doc URL rationale
- `archkit review` — terse `file:line [rule] message`; verbose adds rationale + fix + suppression syntax
- `archkit stats` — terse compact view (existing `--compact` mode becomes default in non-`--json`); verbose adds recommendations rationale

### 7.4 Does NOT apply to

- `--json` output shape (agents parse what they need; verbose mode adds optional fields like `rationale`, `docUrl`, `fix` to each finding/item rather than restructuring the output)
- Existing commands without verbose-aware output (drift, gotcha, init) — keep current behavior; we don't sprinkle `--verbose` across the entire CLI

### 7.5 Output principle

Default output for any agent-facing command should be: **what does the agent need to fix this RIGHT NOW**. Not what does the agent need to understand the problem deeply.

If the agent wants depth, it asks for `--verbose`. The default is action-oriented.

## 9. Error handling summary

Across all new surfaces:

- `--json` mode: `console.log(JSON.stringify({ error: "<code>", ... }))` + appropriate `process.exit()`
- Human mode: `log.error()` / `log.warn()` to stderr + appropriate `process.exit()`
- Logger already writes to stderr — no logger changes needed

Error codes added:
- `unknown_feature` (preflight)
- `unknown_layer` (preflight)
- `weak-suppression` (review — emitted as a finding, not an exit error)
- `incomplete-skeleton` (review — emitted as a finding)
- `existing-claude-hook` (init --install-hooks --claude)
