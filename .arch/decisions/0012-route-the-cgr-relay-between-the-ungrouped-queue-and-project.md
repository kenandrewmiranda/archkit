# 0012. Route the CGR relay between the ungrouped queue and project tracks, with a shared dated queue branch

- **Date**: 2026-06-20
- **Status**: Accepted
- **Tags**: cgr, goals, mcp, git, lifecycle

## Context

ADR 0010 added `project` as a branch-isolated feature set (feat/<project>) and ADR 0011 gave pending goals a queue/ drawer with per-project subfolders. That left the live queue with two distinct tracks: project-grouped goals (each meant for its own feat/<project> branch, worked in parallel) and ungrouped "plain" goals. The relay's nextEligibleGoal silently auto-picked across both, so when a user had both a feature-set in flight AND loose queue work, the relay would guess a track instead of letting the user steer. The source ask: "if there are multiple projects AND items in the queue, ask the user whether they prefer to tackle the queue or the projects. If there isn't a branch established for the current queue, set up cgr-queue-<current-date>." Two sub-problems: (1) routing — surface a choice only when both tracks have ready work, without disturbing the in-progress-resume and depends-on precedence that already govern selection; (2) branch isolation for the ungrouped track — plain queue goals had no branch story (feat/<project> is project-only), so parallel queue work wasn't isolated. archkit's instruct-not-act invariant (ADR 0010) forbids shelling out to git.

## Decision

Add `routeNextGoal(archDir)` as the relay's selection brain, layered over nextEligibleGoal. It returns a tagged result: `resume` (an in-progress goal — always first, never interrupted), `single` (exactly one track has deps-satisfied work — auto-pick via nextEligibleGoal, unchanged behavior), `choice` (BOTH the ungrouped queue and ≥1 project track have eligible work — surface the choice), or `none`. Only deps-satisfied, non-parked, non-completed goals count toward a bucket, so the depends-on gate and in-progress resume keep precedence over the prompt (a blocked or active goal never triggers a choice). The goal_next prompt renders the choice via relayRoutingChoice (presented to the user with AskUserQuestion); the agent then begins the chosen track with the new archkit_goal_start <slug> tool — the actionable companion that marks a SPECIFIC goal in-progress and injects its payload (goal_next itself takes no args, so a specific-goal start needs its own tool). For the ungrouped track's branch isolation, add a single shared dated branch cgr-queue-<YYYY-MM-DD>: ensureQueueBranch records the name ONCE (in .arch/goals/.queue-state.json, gitignored, beside .loop-state.json) the first time an ungrouped goal is started, and every later queue goal in the batch REUSES it. startGoal records it (only for no-project goals); renderPayload reads it purely and emits a branch-prework block mirroring feat/<project> — "git switch -c cgr-queue-<date>" when none is recorded, "git switch <branch>" once it is. The relay renders BEFORE starting so the first queue goal sees create and the rest see switch. The queue-branch block is gated on the parallel regime being active (some live goal carries a project), so single-track projects keep their pre-feature behavior. completeGoal clears the state once the ungrouped queue drains, so the next batch mints a fresh dated branch. archkit only emits guidance and records the name — it never runs git; the date is stamped by archkit. feat/<project> and cgr-queue-<date> are disjoint name schemes and never collide.

## Consequences

Easier: when both tracks have work the user explicitly steers (push a feature set vs drain the queue) instead of the relay guessing; ungrouped queue work is now branch-isolated on one shared dated branch, the queue-side complement to feat/<project>, so parallel agents don't trample the working tree. archkit_goal_start also enables deliberate out-of-order starts. Unchanged/constrained: single-bucket selection, in-progress resume, depends-on gating, and the testing-backlog threshold are all preserved — the threshold is a within-track refinement and does not pre-empt the cross-track choice; instruct-not-act holds (no git is run, branch state is just a recorded name). Back-compat: the queue-branch block only appears when a project goal is live, so existing single-track projects are untouched; .queue-state.json is transient and gitignored. Migration: none — all additive. Covered by 12 new tests in tests/cgr-goals (single/choice routing, resume + deps precedence, branch name derivation + record-once-reuse, create-then-switch payload guidance, scheme non-collision, drain-clears-state, CLI goal start); the new tool is registered in the mcp-server tool-list and silent-success-audit fixtures; full suite green (52/52).
