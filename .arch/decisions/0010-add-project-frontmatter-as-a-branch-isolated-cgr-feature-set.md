# 0010. Add project frontmatter as a branch-isolated CGR feature set, distinct from epic (sequencing)

- **Date**: 2026-06-20
- **Status**: Accepted
- **Tags**: cgr, goals, lifecycle, mcp, git

## Context

ADR 0005/0006 added `epic` as a SEQUENCING group: goals sharing an epic drain one objective fully before the next begins (epic-primary ordering). But the user wants a second, orthogonal grouping for PARALLEL work — relevant CGRs that belong to one feature set and should be developed together on a dedicated git branch, so multiple agents can work different feature sets concurrently without colliding in the working tree. Overloading `epic` to also mean "branch" would conflate run-order with branch-isolation (an epic is about WHEN a goal runs; a branch is about WHERE its edits land). The original ask also floated a Queue folder + per-project subfolders, but ADR 0003/0005 already established that folders-as-meaning is rejected — state lives in `status:` frontmatter and sequencing in `order`/`epic`, not the directory. A branch grouping should follow the same frontmatter-not-folders precedent. Separately, archkit's design invariant (mirrored across boundary/gotcha/graph propose flows) is instruct-not-act: archkit emits guidance text, humans/agents take the side-effecting action. Running `git switch`/`git commit` itself would break that invariant and couple archkit to a VCS.

## Decision

Add an optional `project:` frontmatter field (slugified on write, parsed by parseGoal, emitted by writeGoal/emitFrontmatter — exactly like `epic`) that marks a goal as part of a branch-isolated feature set. `project` is ORTHOGONAL to `epic`: epic = sequencing group (run order), project = branch-isolated feature set (where edits land). When a goal carries a `project`, renderPayload injects a branch-prework block that instructs the agent — BEFORE editing — to `git switch -c feat/<project>` (or `git switch feat/<project>` if it already exists) and to commit each completed CGR to that branch; the block explicitly states archkit only emits the guidance and the agent runs git (instruct-not-act preserved — archkit NEVER shells out to git). `archkit_goal_list` gains a `projects` map (project label -> goal slugs in queue order) mirroring the existing `epics` view, emitted only when some goal carries a project, and listing only real projects (no '(ungrouped)' bucket — ungrouped goals simply don't appear). The `archkit_goal_intake` schema exposes an optional `project` arg whose description distinguishes it from `epic`. `project` does NOT participate in sortGoals ordering — it is a branch-grouping/payload-guidance concern only, so queue order is unchanged and fully backward compatible (absent on every legacy goal).

## Consequences

Easier: a decomposed batch can be tagged into parallel feature sets, each isolated on its own feat/<project> branch, so multiple agents work concurrently without trampling each other's working tree; the `projects` view makes the branch grouping visible at a glance. Unchanged/constrained: queue ordering remains epic-primary (ADR 0006) — `project` is independent of `order`/`epic` and never affects which goal runs next; folders-as-meaning stays rejected (this is frontmatter, consistent with ADR 0003/0005); the instruct-not-act invariant holds — archkit emits branch guidance as payload text and never runs git, so it stays VCS-agnostic and side-effect-free. Migration: none — `project` is additive and absent on existing goals, which render and sort exactly as before. Follow-ups still open from the source ask (NOT in this goal): an actual Queue folder for pending goals, project-completion merge/archive, and an agent chat.md conflict-coordination board — tracked as separate cgr-* goals. Covered by 6 new tests in tests/cgr-goals (project parse/emit round-trip, legacy-untouched, branch-prework payload string present/absent, projects view mirrors epics, projects view omitted when empty); full suite green (52/52 suites).
